# Supabase Cloud Integration Setup

Phase A adds optional machine-status synchronization. It does not start cloud
sync automatically and does not change the controller, game logic, Vision
Service, Game State Engine, or existing REST APIs. Local machine operation is
authoritative and continues when Supabase is unavailable.

## Installation

Install the project dependencies from the repository root:

```bash
python3 -m pip install -r requirements.txt
```

The integration uses the official `supabase` Python SDK. `python-dotenv` loads a
local `.env` file during cloud configuration. The `.env` file is ignored by Git.

## Supabase table

Create the Phase A table in the Supabase SQL editor:

```sql
create table if not exists public.machine_status (
    id bigint generated by default as identity primary key,
    machine_name text not null unique,
    status text not null,
    x_position double precision not null default 0,
    y_position double precision not null default 0,
    claw_power integer not null default 0,
    online boolean not null default false,
    updated_at timestamptz not null default now()
);
```

The service updates by `machine_name` and inserts the first row if no matching
machine exists. Keep `machine_name` unique to prevent duplicate machine rows.
Configure Supabase Row Level Security for the key and deployment model you use.
Never expose a service-role key in browser code or commit it to Git.

## Environment variables

Copy the safe template and add credentials from Supabase project settings:

```bash
cp .env.example .env
```

Required:

```dotenv
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_KEY=your-server-side-key
```

Optional:

```dotenv
CLAW_MACHINE_NAME=Progress Claw Machine
SUPABASE_MACHINE_STATUS_TABLE=machine_status
SUPABASE_RETRY_SECONDS=30
```

## Usage

Cloud integration remains opt-in during Phase A. A future supervisor or runtime
adapter can pass local snapshots to the service without putting Supabase in the
machine control path:

```python
from cloud.sync_service import CloudSyncService

cloud = CloudSyncService()
cloud.connect()
cloud.sync_game_status(
    status="ready",
    x_position=0.0,
    y_position=0.0,
    claw_power=70,
)
cloud.heartbeat()
cloud.sync_online_state(False)
```

Each synchronization returns a `SyncResult`. Callers may inspect it for
monitoring, but must not stop or block local machine operation when `ok` is
false. Failed connections are logged and throttled until the configured retry
window passes.

## Logs

Cloud messages use the `progress_claw.cloud` logger and include these labels:

- `[Cloud] Connected`
- `[Cloud] Disconnected`
- `[Cloud] Sync Success`
- `[Cloud] Retrying`

## Troubleshooting

- **Cloud stays disconnected:** verify `SUPABASE_URL` and `SUPABASE_KEY` are in
  `.env`, then restart the Python process.
- **SDK unavailable:** install `requirements.txt` in the same Python environment
  used to run the service.
- **Permission or RLS error:** confirm the configured key has `select`, `insert`,
  and `update` access to `machine_status` under your Row Level Security policy.
- **Duplicate machine rows:** add the unique constraint on `machine_name` and use
  a stable `CLAW_MACHINE_NAME` per physical machine.
- **Network outage:** no operator action is required for local play. The service
  logs `[Cloud] Retrying` and accepts another attempt after the retry interval.

Game history and AI detections are intentionally outside Phase A.
